# Proposal — make the roadmap a bounded, derived, agent-maintained worklist

Decision needed: this changes how three **human-owned** files are written and how
the guard treats them — `leitwerk/roadmap.md`, `leitwerk/tiers.conf`, and
`leitwerk/constitution.md`. Per the constitution an agent may not edit these; this
stages the exact edits for you to apply by hand (no script) and asks you to accept
the direction. Some edits apply now, some are gated behind code landing — see
**Sequencing**.

## Problem

The roadmap conflates things that should be separate, and it decays because of it:

- **It is used as an append-only ledger** (future + a growing "Recently decided
  (done)" history + full proto-spec detail per item), though its own header
  defines it as "the ordered set of *future specs*". Left unchecked it grows
  without bound — your `M.9999` concern.
- **Ordinal numbers are treated as identity.** Inserting `M2.5` showed the strain;
  the still-open `adopt-superpowers-patterns` proposal wants `M3.8`/`M3.9`, which
  **collide** with any other item numbered the same. An ordinal is not a stable key.
- **Status is written as world-state prose in a guarded file, so it rots.** The
  only party that could keep it accurate cheaply (the agent, from `git`/`gh`/
  `specs/archive/`) is locked out. Concretely: **M2.3 stated "the CI workflow has
  never executed (no remote repo yet)" while the repo was public and the
  `leitwerk gate` workflow had been green since 2026-07-22** — and that stale
  prose then misled an agent this session. A misleading governance doc is worse
  than none.
- **Guarded ≠ verified.** The roadmap is the only governance artifact with a guard
  but no consistency check. `drift`/`lifecycle` cover specs; nothing checks the
  roadmap↔spec link or flags a claim contradicted by reality — which is why M2.3
  rotted silently.
- **The guard is all-or-nothing, so every touch needs a proposal.** That friction
  is what caused the rot: nobody hand-reconciles status through a full
  proposal+apply cycle for a one-word fix.

Root cause: the roadmap drifted from its own definition. The constitution already
has the right policy for the last point — its **Decision routing** section
(lines 54–60) splits *judgment edits* (human-only, from a proposal) from
*mechanical consequence edits* (agent-applicable, audited). The guard just never
mechanized that split.

## Target design

1. **Bounded worklist.** The roadmap holds only *open* intent. An item **leaves**
   the file when it is promoted to a spec (the spec lifecycle tracks it) and its
   durable record lands in `specs/archive/<slug>.md`. Length tracks open work, not
   history — it never reaches `M.9999`.
2. **Slug identity, position order.** The stable key is the **slug** (the token
   after `·`, already present in every title); the milestone number is a human
   label. Specs link by slug, so nothing machine-critical depends on the number
   and reordering never breaks a reference. The superpowers `M3.8`/`M3.9` collision
   dissolves: slugs are unique, position gives order.
3. **Thin items.** One line of intent, an acceptance sketch, a tier estimate. Full
   Problem/Behaviour detail belongs in the spec at promotion, not frozen here.
4. **Status derived, never asserted.** No world-state prose. An item's *acceptance*
   is its status probe: code items derive state from the spec lifecycle
   (`specs/archive/` = landed, active spec = in flight, no spec = not started);
   process items from the existence of their acceptance artifact.
5. **Explicit, checked roadmap↔spec link.** A spec declares `Roadmap: <slug>`;
   `lifecycle` resolves it against `roadmap.md` (orphan = red). This is the `drift`
   pattern one level up. It also fixes the existing `spec-lifecycle`↔
   `lifecycle-check.md` filename mismatch — the link is declared, not inferred.
6. **Two guard classes — mechanizing the split the constitution already states.**
   `[human-owned]` (guarantee files) stays hard-blocked (`deny`; change via
   proposal). `[human-confirmed]` (the roadmap) is agent-editable behind an
   explicit per-edit human confirmation (the hook's `ask` decision). The confirm
   is where you exercise ownership of priority/scope; the thin-item design keeps
   each diff small enough that confirming is real review, not rubber-stamping.

## Design decisions

- **D1 — slug is identity, number is a label.** Kills renumbering breakage and the
  `M3.8`/`M3.9` collision. No header churn needed: the slug already sits in every
  title after `·`.
- **D2 — confirm ≠ authorship.** `ask` is real control only if diffs stay small,
  which the thin-item + items-leave-on-landing design guarantees. Genuine
  priority/scope shifts (a new milestone, a reprioritization) still warrant a
  proposal for the durable record; routine upkeep does not.
- **D3 — sequencing guards a fail-open.** The `tiers.conf` move must not precede
  the guard code. Called out as a hard ordering constraint, not a preference.
- **D4 — derive, don't store status.** No `progress.md`, no `leitwerk status`
  command (both considered, rejected): a stored digest rots the same way. State
  comes from the spec lifecycle + archive + git on demand.
- **D5 — don't churn.** Existing fat items are reworded to thin/durable form on
  the touch that lands them, not in a big-bang rewrite. Only M2.3 (the proven
  offender) is reworded now.
- **D6 — mechanize existing policy.** The two-class guard implements the
  constitution's own mechanical-vs-judgment split; it is not new policy.

## How to accept
Approve the direction. Apply the `roadmap.md` edits now (by hand — you own the
file). The two code items then run through the normal gated flow; the
`tiers.conf`/`constitution.md` edits apply at the steps above. Then delete this
proposal.

## What rejection means
Delete this file; the roadmap keeps mixing intent/history/status behind an
all-or-nothing guard, and status keeps rotting. Partial acceptance is fine — e.g.
take the `roadmap.md` cleanup (1) now and defer the guard class (2–3).

## Recommendation
**Accept in full.** It is the smallest coherent change that closes all five gaps
at once, bounds the file's growth, keeps human ownership of priority intact, and
makes the roadmap honest by construction — consistent with the framework's own
"prose → mechanism" thesis.
