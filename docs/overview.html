Sound familiar?

If you're already building with agents, you have probably thought at least one of these:

“It told me the tests passed. They didn't.”

“I put the rule in CLAUDE.md — the next session ignored it.”

“The plan looked perfect. The code didn't run.”

“My migration shipped with the same check as a typo fix.”

“I changed the code; the spec still describes the old behaviour, and nothing flagged it.”

“The feature shipped two weeks ago — my spec's still on draft.”

Different symptoms, one cause: nothing outside the agent was checking whether the result was actually correct — or whether the spec still matched the code. That is the gap this closes.

AI agents produce code faster than anyone can review it, so the bottleneck moves from writing code to knowing whether it is correct. Two facts make that hard. First, self-evaluation is unreliable: an agent asked to grade its own work can report a success it cannot substantiate — the whitepaper cites the evidence. Second, manual review does not scale to the volume agents generate. The accept/reject decision therefore has to come from outside the agent — from something its own reasoning cannot override.

The problem

Why a research → design → plan pipeline isn't enough

A phase pipeline organizes how the agent works — research, design, planning — each step producing prose that is judged, by the agent or a reviewer, at the end. That structures generation; it does not constrain correctness. In practice:

  • ✗A convincing plan and a broken result can coexist. The plan says nothing about whether the code that follows is correct.
  • ✗Self-assessment is unreliable. An agent grading its own work can report "tests pass" without being able to substantiate it.
  • ✗Manual diff review does not scale to the volume agents generate.
  • ✗The prose artifacts accumulate and fall out of date. Research, design, and plan documents are advisory, so they are skipped under context pressure.
  • ✗Spec and code diverge with no mechanism to detect it.

The common root: the decision to accept rested on a judgment — the agent's or a reviewer's — that the agent's own reasoning could work around.

The shift

Put an external check at the center

Keep lightweight phases — spec → plan → build → review — but make the center a check, not the sequence of thinking steps. Every change must pass it before it lands:

$ leitwerk verify --tier <T0|T1|T2> # exit 0 → it lands · anything else → blocked

The check returns an exit code from real tools — compiler, tests, SAST, structural checks. Three properties follow from it being a plain command rather than an instruction:

Non-bypassable

The result comes from tools, not from agent self-evaluation. The agent cannot modify the gate to make itself pass.

Runtime-independent

CI runs the same command with no agent present, so the bar never depends on which agent — or which model — produced the change.

One source of truth

The same command is the authority everywhere; the bar cannot diverge between what the agent ran and what CI runs.

What it runs

What the gate actually does

verify runs a set of checks chosen by the change's blast-radius tier — cumulatively, so a T2 change runs everything T0 and T1 run, plus more. Each check wraps a real tool and returns pass, fail, or skip (nothing to run — an honest abstention, never a faked pass). A single failure makes the gate red.

Check Verifies T0 T1 T2
lintStyle and static conventions●●●
typesType / contract check — the cheapest oracle●●
testsYour test suite — the behaviour oracle●●
driftSpec ⇆ code divergence●●
sastSecurity static analysis + dependency policy●
erosionComplexity / duplication ceilings●

The tier is not a judgment call — verify --auto derives it from the changed paths: a docs edit is T0, an application change T1, anything under db/migrations/ or infra/ is T2. You map paths to tiers once, in a human-owned policy file; the agent cannot narrow its own scope.

The result is a plain exit code. Green lands; red goes back to the agent:

$ leitwerk verify --auto
auto: tier T1 (api/orders/service.go); 3 changed files
leitwerk verify (tier T1)
checks: lint types tests drift

  ✓  lint     0 issues
  ✓  types    ok
  ✓  tests    128 passed
  ✓  drift    in sync

gate: PASS   exit 0 → it lands
$ leitwerk verify --auto
auto: tier T2 (db/migrations/012_refunds.sql); 6 changed files
leitwerk verify (tier T2)
checks: lint types tests drift sast erosion

  ✓  lint     0 issues
  ✓  types    ok
  ✗  tests    2 failing (refund_test.go)
  ✓  drift    in sync
  –  sast     skipped (no scanner wired)
  ✓  erosion  within budget

gate: FAIL   exit 1 → blocked

The difference

But don't CLAUDE.md and rules already handle this?

Teams already steer agents with a project CLAUDE.md, .claude/rules/, skills and slash commands, PR templates, and agent-driven review. Leitwerk keeps all of them — they are how the agent knows what to do, and the natural home for team conventions. But each one is an instruction the model chooses to follow: under context pressure it can skip a rule or leave a template unfilled. They shape behaviour; they do not guarantee it. The gate is the one piece that turns should into must — it runs the same tools every time and returns a verdict the model cannot reinterpret.

Mechanism What it's good for Why it isn't the guarantee
CLAUDE.md & project rules Standing conventions, safety notes, and pointers, loaded into every session. Advisory — honoured at the model's discretion and squeezed first under context pressure. Even keeping them lean is a convention unless a check enforces it.
Skills & slash commands Repeatable procedures — create a PR, run a review, research the codebase. Run only when invoked, and their internal "done" / "tests pass" claims are self-reported.
PR & issue templates A consistent shape for human-readable output. Purely advisory — routinely left unfilled or reordered, and nothing stops a merge when they are skipped.
Agent-driven review Surfaces findings, severity, and a report to triage. Findings are advisory, and the reviewer is another model grading model output — a human still has to decide.
The gate — leitwerk verify The accept/reject decision itself. — it is the guarantee: deterministic, external, non-bypassable.

So the advisory layer and the gate are not alternatives. Rules, skills, and reviews make the agent productive; the gate makes the result trustworthy. Leitwerk's own rules and review workflow are advisory too — they feed the gate, they do not replace it.

Around the gate

Who owns what

You own intent and judgment

Requirements, the spec, and the review only a human can do. Just three kinds of decisions escalate to you: intent and priorities, anything that weakens a guarantee, and accepting irreversible risk. Everything else the agent decides and records.

The agent owns generation

Most of the code, guided by the spec — and the routine decisions it can keep reversible.

Spec and code stay bound

They co-evolve. A change that makes them disagree is surfaced as drift and left for a human to reconcile — never silently resolved.

Durable output has one home

What lasts lands in the spec, the plan, or the decision record — no parallel tree of prose documents to fall out of date.

Scrutiny scales with blast radius — how much damage a change could do if it's wrong:

T0 Display / low-risk code. Light checks.
T1 State-changing code. More checks.
T2 Irreversible / infra / data / auth. The most, plus human sign-off.
Enforced in two places. While you work, a hook in the agent stops it from finishing a task while the check is red. Before code ships, CI blocks the merge. Same command, same bar, both times.

Generality

Tool-agnostic by construction

The command and tier names above are one instantiation. The method itself doesn't depend on a particular agent: the gate is a plain executable, so the same guarantee holds whether the code came from Claude Code, an open-code agent (Codex, Copilot, Cursor, Aider…), or a human — and CI enforces it identically. Adopt the gate first; the agent tooling is optional ergonomics layered on top.
