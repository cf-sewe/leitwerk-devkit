Technical Whitepaper · AI-Native Software Engineering

Leitwerk: A Framework for AI-Native Application Development

Spec-anchored, gate-centric, role-based delivery for software built and maintained primarily by AI agents — across greenfield, feature, and bugfix work.

Version Draft 0.1 Date 2026-07-19 Scope Tool-agnostic core · Claude Code & open-agent bindings Design assumption Opus-4.8-class agentic capability

Abstract

AI coding agents at current capability (Opus-4.8-class and above) can generate and modify code faster than humans can read it, but the empirical record indicates that without structure, quality erodes. Roughly 45% of AI-generated code fails security tests12, iterative self-refinement increases critical vulnerabilities13, code duplication rises while refactoring falls14, and developers systematically over-estimate their own speed-up21. The evidence also shows what works: treat the model as a cheap, fallible generator and place correctness decisions in an external gate whose policy the agent cannot change unilaterally76; retrieve context rather than pre-loading it17; and keep humans on the decisions agents demonstrably get wrong.

Leitwerk operationalizes these findings. It rests on three layers — durable git-versioned artifacts, an ensemble of specialized agent roles, and a deterministic verification gate — organized so that humans own only requirements and behavioral review while everything between is machine-enforced and tiered by blast radius. This document defines the problem, the principles, the roles and how they are invoked, capability-tiered model routing, three concrete workflows (greenfield, feature, bugfix), the gate, the human-in-the-loop model, and adoption bindings for Claude Code and open agent tooling. Leitwerk is spec-anchored rather than regenerate-only, so a human expert may still write code by hand; divergence between specification and code is surfaced for a human to reconcile rather than silently overwritten.

On the name. Leitwerk (German) is an aircraft's tail assembly — the control surfaces that keep a fast, powerful machine stable and on course. The framework is the leitwerk around a powerful model: the model provides thrust; the framework provides control and direction.

1Introduction & motivation

Leitwerk's goal is a rigorous, scalable process for software that is written mostly by AI and reviewed — never hand-read line-by-line — by humans. Humans contribute requirements and judge results; agents do the construction.

1.1 The opportunity, and why structure is required

A capable agent can now implement a feature end-to-end, and cheaper models can perform routine work — locating code, summarizing logs — at negligible cost. The temptation is to point an agent at a task and let it run. The evidence indicates this fails at scale: security defects are common and get worse under iteration13, duplication accumulates while refactoring declines14, and long-running agents lose track of their own goals as context degrades17. The value of a framework is precisely to convert raw model capability into sustained quality.

1.2 The problem to solve

The one-shot ceiling

Applications of real size cannot be produced in a single pass. Naive iteration compounds errors rather than converging — an agent's own past mistakes condition its future ones.18

Requirement drift

Even when specified up front, requirements are hard to keep current and harder to keep enforced. Code and intent silently diverge until the system no longer does what was asked.

Quality erosion

Left ungated, AI-authored codebases trend toward duplication, shallow tests, and undetected vulnerabilities — and the decline accelerates as the codebase grows.14

Context limits

No context window is large enough to hold a real system reliably; models degrade well before the limit, so understanding must be retrieved, not memorized.17

1.3 The three use cases

Leitwerk treats three practical situations as first-class; each is a concrete workflow in Section 8.

A · Greenfield

A new, complex application built from nothing — too large to one-shot, must grow over many sessions without losing coherence.

B · Feature on existing

A new capability added to an existing, possibly legacy codebase the agent did not create and does not fully understand.

C · Bugfix

A defect in existing software — localize, reproduce, fix, and prove no regression, at the lightest rigor the change's blast radius allows.

1.4 Goal, non-goals, and the design assumption

Goal: flat-or-improving quality as an AI-authored codebase grows, with humans limited to requirements and behavioral review. Non-goals: Leitwerk is not a new IDE or model; it is not tied to any stack; and it does not require full regenerate-only development, so a human expert may write code by hand.

Design assumption. Leitwerk assumes agents of Opus-4.8-class capability for reasoning-heavy work — architecture, ambiguous specification, adversarial verification — and uses smaller, faster models for routine work such as locating code and summarizing (Section 6). The framework is not a model; it is the scaffolding that makes a capable model reliable and a cheap model useful.

2Background & evidence

Each of Leitwerk's choices traces to a measured result in the literature (2023–2026). The findings cluster into five themes.

How to read this evidence. Two kinds of claim appear below. Model-capability snapshots — failure rates, context-length thresholds, benchmark scores — are tied to the model generation tested; several were measured on GPT-4-class or early-2025 models and will drift as models improve, so they are dated in §14 and should be re-checked, not read as current. Structural, methodological, and human-factors findings — verification-before-acceptance, spec enforcement, characterization tests, automation bias — are largely model-independent and durable. Critically, the snapshots do not prop up the framework: because Leitwerk is defensive, better models lower the frequency of these failures without removing the asymmetric cost of the rare one, so the gates remain relevant whichever way the numbers move. The framework's own cost is measured, not assumed (§9.4).

2.1 Verification is the bottleneck, not generation

The strongest systems constrain control flow and gate acceptance on an external oracle. Agentless — a fixed localize→repair→validate pipeline with no autonomous tool-choosing — topped the open-source SWE-bench Lite leaderboard as of mid-2024 at a fraction of the cost6. Clover reduced correctness to a consistency check among code, docs, and formal annotations and, with GPT-4 on the Dafny CloverBench, accepted correct programs at high rate with zero false positives on incorrect ones7. AlphaCodium's designed test-generate-run-fix flow more than doubled the pass rate of GPT-4 in early 20248. The specific models and rankings have since moved on; the durable lesson — a designed pipeline with an external check beats open-ended self-direction — is what Leitwerk builds on.

2.2 Quality erodes without gates

Across 100+ models tested in 2025, ~45% of AI-generated samples failed security tests, with no improvement as models scaled12. · Critical vulnerabilities rose +37.6% after five refinement iterations on 2025-era models13. · Duplicated blocks rose ~8→12% while refactoring fell from 25% to under 10% of changed lines, across 211M lines committed 2020–202414. · ~1 in 5 packages an LLM suggested did not exist ("slopsquatting"), across 16 models in 202515.

These are 2025-era snapshots and the exact rates will fall as models improve — but that does not weaken the argument, because none of the gates that follow depend on a particular number: an external security check, a dependency allowlist, and duplication budgets cost the same whether the failure rate is 45% or 5%; whether that fixed cost is justified depends on the defect costs avoided, which the evaluation plan (§12) proposes to measure rather than assert. The load-bearing conclusion is structural, not numerical: self-review is not a gate — a reviewer that shares the author's blind spots, or is the author, cannot be the thing that certifies the work — so the durable guardrails are the ones the generating agent cannot author or weaken for itself.

2.3 Long-horizon degradation is intrinsic

Long context is not free accuracy. On the frontier models tested in mid-2025, one analysis found retrieval degrading well before the window filled — around 50K tokens even in a 1M-token window17 — echoing the U-shaped position bias documented on 2023-era models16. That specific threshold is model-generation-bound and should not be read literally for Opus-4.8-class models, which handle long context far better; it is cited as motivation, not as a fixed limit. The durable point does not expire: an agent's own errors can compound over a long session18, so the robust design keeps state in external, structured memory with cold-start recovery as the default19 and retrieves over a structural repo map rather than stuffing context9 — a trade-off that currently favors retrieval on cost and accuracy, and that should be re-checked as context windows improve (§13).

2.4 Specifications must be enforced, not merely written

Spec-driven development has a maturity ladder — spec-first, spec-anchored, spec-as-source1. Production practice lands on spec-anchored: specs and code co-evolve, bound by executable checks. Even Amazon found tests alone insufficient to capture intent and re-added prose specs — but as context, not enforcement3. Documentation and specs rot; the countermeasure is provenance labeling and anchoring so drift fails CI rather than misleading silently1110.

2.5 Human review is valuable but fragile at scale

Humans reliably catch intent, edge cases, and missing context, but AI code "looks clean even when wrong," and automation bias raises the rate of following bad advice by ~26%22 — a human-factors effect independent of model generation. A controlled study on early-2025 tools even found experienced developers 19% slower with AI while believing they were ~20% faster; the authors later revised the study's design (Feb 2026), so the magnitude is contested and tool-bound — but the perception gap it exposed is the durable, design-relevant finding21. The scalable response is well-established: Meta's RADAR uses a risk-tiered funnel that auto-lands low-risk diffs and concentrates human attention on high-risk ones, auto-approving ~60% while keeping revert and incident rates a fraction of manual review23. And for existing code, two model-independent practices hold regardless of capability: pin behavior with characterization tests before change24, and wrap generation in build+test validation loops, as large-scale migrations do25.

3Design principles

Nine principles follow directly from Section 2. They are the core; everything else is implementation.

P1 · Cheap generation, strict acceptance

The model generates freely; correctness is decided by an external gate it cannot weaken.7

P2 · Enforce with executables, not prose

Requirements are kept honest by tests, contracts, and CI checks. Prose is context.3

P3 · Pin behavior before changing it

Code the framework did not generate under gate gets characterization tests first.24

P4 · Retrieve, don't preload

A structural repo map is always available; the context window stays small and high-signal.17

P5 · Cold-start recovery is default

Durable artifacts on disk carry all critical state; no session is irreplaceable.19

P6 · Tier rigor by blast radius

Read-only code is checked lightly; irreversible code demands safety proofs and sign-off.

P7 · Review behavior, surface drift

Humans judge specs, tests, and rendered results; spec↔code divergence is escalated, not auto-resolved.21

P8 · Budget against erosion

Duplication, complexity, and churn have hard ceilings; scheduled consolidation runs.14

P9 · Humans own architecture

Agents do local/mechanical change well and system-wide reasoning poorly; keep humans on decomposition and seams.26

4Architecture: three layers

The central move is to separate what the agent drives from what it cannot skip or edit. Three layers realize this (Figure 1). Authority flows down: artifacts define intent, roles produce work, and the gate decides what may be committed. A feedback edge lets work update the artifacts — but only after passing the gate.

Layer 1 — Durable artifacts. The source of truth, versioned with the code. The constitution (small, human-owned) states architectural principles, the blast-radius tier definitions, mandatory gates, and hard boundaries. Living specs hold behavior as Given/When/Then, each fragment anchored to the code and tests that satisfy it. The decision log (ADRs) records the why so later sessions do not re-litigate settled choices20. The progress / feature list starts every planned feature marked "failing" to prevent premature completion19. The repo map is a tree-sitter + PageRank symbol map9, extended to a typed code graph for large repositories.

Layer 2 — Roles. Specialized agents, each with scoped context and an assigned model tier (Sections 5–6).

Layer 3 — Gate. A single deterministic command, leitwerk verify, composed of tier-selected checks (Section 9). It is the operational form of P1–P2: its verdict comes from tools, not from any role's judgment.

As implemented in this repository. Layer 1 is the tracked leitwerk/ tree, not an abstraction: the human-owned constitution.md (invariants, blast-radius policy, Definition of Done, and the decisions-of-record — there is no separate ADR file), tiers.conf (the blast-radius map: tiers→checks, paths→tiers, and the [human-owned] list), the ordered roadmap.md (the backlog of future specs), and specs/ (living specs). Layer 3 is the compiled gate at core/bin/leitwerk. A repository is brought under the framework by leitwerk init, which scaffolds the constitution and tiers file (plus the Claude steering files of §11); specs and a roadmap accumulate through the process itself.

A living spec states the observable contract, not the implementation, so a reviewer can tell what "correct" means without reading the diff. This repository governs itself, and its own gate is anchored to leitwerk/specs/self.md (excerpt):

# Spec — leitwerk-devkit governs itself

## Behaviour (the observable contract)
- `leitwerk verify` runs from the repo root using `leitwerk/tiers.conf` and the
  repo-local checks in `leitwerk/checks/`, falling back to built-in `core/checks/`.
- Changing a `core/bin`, `core/checks`, or any `*.sh` path selects tier T2.
- A T2 gate runs: `json` (all manifests parse), `shell` (`bash -n` + shellcheck
  when present), `drift` (specs tracked), `selftest` (builds the Go CLI, runs its
  unit + integration tests, re-asserts the external contract, and executes the
  documented scenarios in `examples/scenarios/`), `parity` (the guarantee stays
  in `core/`, stdlib-only; bindings delegate).

## Acceptance checks
`leitwerk verify --tier T2` exits 0 on the clean repo, and exits non-zero if any
of: a manifest is corrupted, a script has a syntax error, or a tier boundary in
`selftest.sh` regresses.

Roles operate a single repeating loop — spec → plan → build → verify → review — shown in Figure 2. The gate is a hard stop: a failing gate returns work to the builder; a passing gate reaches human review. Feedback from review, or a hand-edit by an expert, flows back to the specification rather than being applied blindly.

6Capability-tiered model routing

Matching model capability to task difficulty, rather than applying one model uniformly, aims to improve cost and reliability together. The architect/editor split — a strong model plans, a cheaper model applies edits — reached state-of-the-art at lower cost in its authors' account9 (practitioner-reported, not a controlled study), and multi-agent systems find that token spend explains most of the variance in outcome19, so spend must be directed deliberately.

Task classExamplesModel tier
Architecture & ambiguitysystem design, seam-finding, spec formalization, tier assignment, conflict resolutionOpus 4.8+
Adversarial verificationcorrectness & security review, drift adjudication, "is this really done?"Opus 4.8+
Complex constructionnon-trivial business logic, cross-file coordinated editsOpus 4.8+
Structured constructionboilerplate, mechanical refactors, format-driven edits, test scaffoldingSonnet-class
Retrieval & summarizationlocate code, gather references, condense logs / diffs, first-pass triageHaiku-class

Routing is a property of the role (its default tier) and may be overridden per task by the Orchestrator when a "structured" task turns out to carry hidden reasoning. The principle: reserve the expensive model for judgment; buy breadth cheaply.

7How roles are invoked

Roles are triggered three ways, in priority order.

Phase-driven

The Orchestrator invokes roles per workflow stage: Architect during design, Implementer during construction, the verification panel at the gate. This is the default backbone of every workflow in Section 8.

Signal-driven (the trigger table)

Observable properties of a change are meant to wake the right specialists without manual routing — a mechanism adapted from "codified context" trigger tables10. In the current implementation only the path→tier mapping is mechanical; the diff-signal rows below are carried as review instructions in the skills and constitution, not as automatic triggers.

Human-summoned

A human can invoke any role directly ("have the security specialist look at this") at any point — the escape hatch that keeps humans in control without reading code.

Realization: native orchestration where the platform provides it

The ensemble and its trigger table are the tool-agnostic design; each platform realizes them with its own mechanism. On Claude Code, this layer is not a bespoke orchestrator role — it is a dynamic workflow: a deterministic orchestration script that fans roles out in parallel and has independent agents attempt to refute each finding before it counts. The mapping: the orchestrator becomes the workflow script (control flow in code, not model discretion); the roles become the workflow's agentTypes, reused unchanged; the trigger table is realized so far only as the workflow's tier-driven fan-out — the diff-signal triggers of the table above remain a design target. Two properties bound this. First, workflows are opt-in and scaled by blast radius — a full adversarial panel runs for T2, while T0/T1 stay lightweight. Second, the workflow's verification is soft (agents judging agents, better because it is adversarial and diverse, but still model judgment); it never replaces the hard gate. leitwerk verify remains an external oracle the workflow cannot overrule, enforced at turn end regardless of how the panel voted. Platforms without such an engine (open-code / Codex) run the roles sequentially and lean on the same gate in CI — the guarantee is identical because it lives in the gate, not the orchestration.

Concretely, the workflow is .claude/workflows/leitwerk-review.mjs. It runs three phases — Review (one role subagent per dimension, scaled by tier: correctness at T0, +spec-fidelity at T1, +security at T2), Verify (independent agents try to refute each finding; unrefuted findings survive), and Gate (it runs leitwerk verify and only reports the result). Its return value marks a change landable only when the gate is green and no finding survived — but the gate, enforced at turn end, is what actually blocks. A Claude Code plugin cannot package a workflow, so it ships as a core/templates/ scaffold that leitwerk init writes into the repo; the phase skills prefer it and fall back to spawning the roles directly, so review never depends on it being present.

8Workflows: greenfield, feature, bugfix

The three use cases share the core loop — spec → plan → build → verify → review — but enter it differently and weight the bands differently (Figure 4). Greenfield leads with architecture; a feature on existing code must first onboard and pin behavior; a bugfix optimizes for the lightest safe path.

8.1 Workflow A — Greenfield complex application

  1. Constitution interview. Architect + human. The one heavy human-input moment: stack, principles, tier rules, boundaries. Seed one high-quality exemplar per code type so the agent matches its style thereafter.2
  2. Architecture & decomposition. Architect (Opus 4.8+). Design modules and seams; write initial ADRs; produce a feature list with every item marked "failing."
  3. Per-feature loop. Spec Steward → Scout → Implementer. For each feature: write the spec, retrieve only the relevant context via the repo map, implement in small tiered tasks.
  4. Gate. leitwerk verify + verification panel. Tier-selected checks plus the woken specialists must pass before the feature flips to "passing."
  5. Review checkpoint. UX / Visual Reviewer → human. Human approves behavior and rendered result, not diffs. Feedback re-enters at the spec.
  6. Consolidation. Implementer (mechanical tier). Scheduled de-duplication and dead-code passes keep erosion budgets green as the app grows.

8.2 Workflow B — Feature on an existing (possibly legacy) codebase

The difference from greenfield is a mandatory onboarding prefix, because the agent did not create this code and must not trust its own assumptions about it.

  1. Index, don't ingest. Scout (small) + tooling. Build the repo map / code graph so retrieval, not context-stuffing, drives understanding.9
  2. Reverse-derive specs with provenance. Spec Steward (Opus 4.8+). Extract behavioral specs for the affected area, labeling each claim CONFIRMED / INFERRED / GAP; INFERRED and GAP require human validation before they gain authority.11
  3. Pin behavior. Test Engineer. Generate characterization tests over the area about to change — the regression net that makes the rest safe.24
  4. Seam-bounded design. Architect + human. Find seams; plan the change as small, reversible, independently verifiable units (strangler-fig discipline), keeping the legacy path as fallback.27
  5. Loop + gate + review. As in Workflow A, with the characterization tests joining the gate so existing behavior cannot silently break.
Why this order. Raw LLM spec-extraction scores only ~0.64 F1 unaided11, and characterization tests can lock in existing bugs as "correct." Provenance labeling and human validation of INFERRED claims are what make brownfield onboarding trustworthy rather than confidently wrong.

8.3 Workflow C — Bugfix on existing software

Optimized for speed at the lightest rigor the blast radius allows (P6), while still proving no regression.

  1. Reproduce & localize. Scout (small) + Correctness Reviewer. Reproduce the defect, then localize it via the code graph — structural navigation beats semantic search on hidden dependencies.28
  2. Pin, then fail. Test Engineer. Write a failing test that captures the bug plus a characterization test around the touched code.
  3. Fix at the right tier. Implementer. Apply the minimal fix; the change's blast radius (T0/T1/T2) selects the gate and which specialists wake.
  4. Gate = regression + tier checks. leitwerk verify. The new test passes, the characterization tests stay green, erosion budgets hold.
  5. Review proportional to risk. Human. A T0 fix with a green gate may auto-land or be sampled; a T2 fix (auth, data, money) requires sign-off.23

9Verification & the gate

leitwerk verify is one deterministic command, composed of tier-selected checks (Figure 5). The agent runs it locally; CI runs it authoritatively; on Claude Code it is also wired as a Stop hook, so a turn cannot end on a red gate at the hook's configured tier (the hook reads $LEITWERK_TIER, defaulting to T1 — per-change tier selection from the diff happens in CI, not in the hook). Gate definitions are human-owned — the tier→check policy lives in the tiers file, governed by the constitution: an agent may propose a new check but cannot lower a threshold, remove a check, or downgrade a path's tier without a human-approved change to those files.

TierExamplesGate (cumulative)Human checkpoint
T0 read-only / displayrendering, formatting, read queries, loggingcompile/types · lint · SAST on diff · dependency allowlist · erosion budgetsvisual / functional review; may be sampled or auto-landed
T1 state-mutatingwrites, business logic, workflows, non-destructive APIs+ property & contract tests · state invariants · mutation-score floor · behavior-level acceptance testsspec + behavior review
T2 irreversible / infradeletes, migrations, money, permissions, auth, infra/config+ explicit safety assertions on the destructive op · dry-run / rollback proof · characterization tests greenmandatory human sign-off

Two cross-cutting checks are part of the design. The drift sensor is specified to confirm spec fragments still resolve to the code and tests they are anchored to, surfacing unresolved anchors for a human to reconcile (never auto-resolved) — the shipped check is currently a placeholder that only counts tracked specs; the anchor mechanism is roadmap item M1.1 and an open question in §13. An LLM-as-judge / panel review runs as an advisory annotation only — never the sole gate, because same-model reviewers share the generator's blind spots29.

Mutation score, not line coverage. AI-written tests frequently achieve high coverage while asserting nothing meaningful. A mutation-score floor — deliberately injecting faults and requiring the suite to catch them — is what proves the tests are load-bearing rather than decorative.

9.1 The gate as a command

The gate is a single static binary — core/bin/leitwerk, a compiled Go program (source: core/cmd/leitwerk; the binary itself is a build artifact, not committed) with no third-party dependencies, so it cross-compiles to one file and runs with only a shell present. The checks it orchestrates stay as shell scripts that shell out to the real toolchains; the binary only selects and runs them. The same binary is what CI runs, what a Claude Code Stop hook runs, and what an open-code agent runs from AGENTS.md — so the strongest guarantee never depends on which agent produced the change.

SubcommandWhat it does
verify [--tier T0|T1|T2]Run the tier's cumulative checks; print a line per check and a PASS/FAIL verdict.
tier <path>Print the blast-radius tier for a changed path (first-match glob table).
guard <path>Exit 3 if the path is human-owned; 0 if editable.
driftSurface spec↔code divergence (never resolves it).
init [dir]Scaffold leitwerk/{constitution.md,tiers.conf} and the Claude steering files.
versionPrint the version.

Exit codes are the contract every surface keys off: 0 gate green / path editable · 1 a check failed (gate red) · 2 usage error · 3 path is human-owned. A single check's own exit 2 means skip (nothing to run) and never fails the gate — a green gate means checks ran or abstained visibly. For the shipped checks that property holds by construction; for repo-local overrides it is a constitutional convention review must uphold — the gate cannot verify a check's own honesty.

9.2 The policy file, and the shipped defaults

The tier table above (Fig. 5) is the design target a mature repository grows into; the shipped defaults are leaner and auto-detect a toolchain, skipping cleanly until wired to a project's real build/test/SAST. Policy is one human-owned file, leitwerk.tiers (an adopter's copy is leitwerk/tiers.conf), with three tables: [tiers] maps a tier to its cumulative check list, [paths] maps a path glob to a tier (first match wins), and [human-owned] lists the files an agent may propose but not edit.

Each check is one script that exits 0 (pass), 1 (fail), or 2 (skip). The built-in checks in core/checks/ auto-detect a toolchain: lint, types, tests, sast (semgrep when installed; a dependency-policy check is a design target, not yet shipped), drift (placeholder, see above), and erosion (duplication budget when an analyzer is installed). A consuming repo overrides any check by dropping its own <name>.sh into leitwerk/checks/ — it never edits the installed core. This repository does exactly that: being a polyglot tool rather than an app, its own T2 gate is json shell drift selftest parity, with repo-local checks in leitwerk/checks/ — json (manifests parse), shell (bash -n + shellcheck), selftest (the CLI's golden suite: it builds the Go binary, runs its unit + integration tests, re-asserts the external contract, and executes the runnable scenarios in examples/scenarios/ — each an executable statement of one framework guarantee), and parity (the boundary fitness function of §11), plus context (§9.4 — budgets the always-on context surface; runs at every tier). 9.3 The human-owned guard

Gate definitions are human-owned, and that boundary is enforced mechanically rather than by prose. leitwerk guard <path> reports whether a path matches the [human-owned] list; the Claude binding wires it to a PreToolUse hook (leitwerk-hook-guard) that blocks a Write/Edit to the constitution, the tiers file, or the roadmap before it happens — open code enforces the same boundary with required review / CODEOWNERS. An agent proposes changes to these files; a human makes them. This is a guardrail: the hard guarantee is still the gate, so open-code parity is unaffected. The guard's mechanical scope is deliberately narrow — it protects the three policy files, not the check scripts, which remain agent-editable at tier T2 (reviewed, and covered by the CLI's own golden suite in this repository). The threat model in §13 discusses what that boundary does and does not defend against.

9.4 Cost of the framework itself

A framework that argues from token economics19 must account for its own overhead. Three cost surfaces exist, and each is either measured or bounded here (figures from this repository, 2026-07-19; they are snapshots and re-measurable at any time).

  • Always-on context. What every agent session pays before doing any work: the scaffolded CLAUDE.md (≤ 200 lines by template guidance) plus the skill/agent frontmatter descriptions the session's system prompt lists. Measured on this repository: ≈ 990 estimated tokens total (bytes/4 heuristic). Everything else — the constitution, skills' procedure bodies, role charters, specs — loads on demand: skill bodies are 171–315 words, role charters 115–166 words, and a session reads only the specs relevant to its change. This split (small always-on bridge, procedures on demand) is the direct application of P4 to the framework's own artifacts.
  • Gate execution. Wall time, not tokens: on this repository (warm build cache) T0 ≈ 0.1 s, T1 ≈ 0.5 s, T2 ≈ 1–2 s including compiling the Go CLI and running its test suite; the verify output an agent reads back is ≤ ~420 bytes. A red gate feeds only the failing check's last line into context, not a log dump.
  • Review fan-out. The T2 workflow's cost is bounded by construction: at most 3 role reviews, at most 8 refutation agents per dimension (a schema-level cap), and 1 gate runner; T0/T1 skip the panel entirely. This is the largest token cost in the framework and it is paid only at the highest blast radius.

The always-on surface is also gated: a repo-local context check budgets CLAUDE.md (200 lines), each path-scoped rule (100 lines), each frontmatter description (80 words), and the always-on total (2 000 estimated tokens), so a change that bloats the per-session tax turns the gate red instead of degrading every future turn silently. Outcome-level efficiency — tokens-to-green-gate, human interventions per landed change — is stochastic and model-bound; it belongs in the periodic evaluation harness of §12, not in a per-change gate.

10Human-in-the-loop

Humans own exactly two things: the constitution and specs (intent) and the behavioral / visual review (acceptance). Everything between is machine-gated. Three rules keep oversight cheap without losing safety.

Review behavior, not diffs

Reviewers see rendered screenshots and visual diffs (approve / reject / ignore, grouped by cosmetic / functional / critical), acceptance scenarios in business language, and the drift report — the artifacts humans judge well.

Risk-tier and prune on data

Start with checkpoints at every high-risk step; track approval and revert rates; remove checkpoints where humans approve ~99% and never modify. This is the RADAR funnel that auto-lands ~60% of diffs safely.23

Design against over-trust

Self-reported confidence is unreliable21. Drive checkpoint pruning from objective signals — revert rate, incident rate, regression escapes, per-cohort pass rates — not from how fast the work feels.

Bidirectional refinement

Feedback after review, or an expert's hand-edit, updates the spec: the Spec Steward proposes a reverse-derived spec change (labeled INFERRED) for human confirmation. The spec stays authoritative; the code is allowed to lead when an expert wants it to.

11Adoption: Claude Code & open agents

Leitwerk's core — the artifacts, the gate, and the tier policy — is tool-agnostic and lives in core/. Each tool gets a thin binding under bindings/ that invokes the core and never reimplements it. The leitwerk verify gate is a plain executable wired into CI, so the strongest guarantee never depends on a particular agent runtime.

The two bindings are deliberately not symmetric, and the framework is explicit about where the asymmetries fall:

  • A plugin cannot package everything. CLAUDE.md, .claude/rules/, and the review .claude/workflows/ are repo-level Claude Code primitives a plugin cannot carry, so leitwerk init scaffolds them from core/templates/ instead.
  • The marketplace sparse-clones only the plugin subdirectory, so the plugin's launcher cannot reach a sibling core/; an adopter supplies the core binary via LEITWERK_HOME or a global go install.
  • Guarantee-parity, not ergonomics-parity. The hard guarantee — the gate, reachable via leitwerk verify + CI with no agent runtime present — must hold identically on every tool. The ergonomics (Claude Code skills, subagents, the fan-out workflow) are not expected to reach open code, and that is acceptable, because they never carry the guarantee.

The last point is enforced structurally, not by convention. A structural fitness check — leitwerk/checks/parity.sh, part of this repo's own gate — fails if gate logic leaks into a binding, if a binding ships checks or tier policy, or if the core gate acquires a third-party dependency; and a dedicated CI job runs the core gate with no agent runtime and without the optional analyzers, proving an open-code adopter gets the same result. If that job is green, the guarantee has reached open code whether or not any of the Claude Code ergonomics did.

12Evaluation plan

Leitwerk should be judged on sustained delivery, not one-shot benchmarks — headline SWE-bench numbers are contaminated and skewed toward trivial fixes30; long-horizon, multi-file suites (SWE-bench Pro, SWE-rebench) are the more informative signal. Proposed rollout and metrics:

  1. Phase 0 — the gate first. Build leitwerk verify (tiered DoD + drift sensor + erosion budgets) and wire it into CI on one existing application. Highest leverage, best evidence, immediate value before any role exists.
  2. Phase 1 — greenfield loop on one new small application end-to-end.
  3. Phase 2 — brownfield adoption (repo map, provenance spec extraction, characterization tests) on one existing application.
  4. Phase 3 — long-lived hardening: consolidation passes, checkpoint pruning from telemetry, growing golden suite with per-cohort monitoring.

Metrics. Revert rate and production-incident rate of AI-landed changes (vs. a manual baseline); regression escapes caught by the golden suite; mutation score trend; duplication / complexity / churn against budget; human review minutes per landed change; drift incidents surfaced vs. shipped. Success is flat-or-improving quality metrics as the codebase grows — the erosion the evidence predicts, not observed.

Efficiency. Two measurement classes with different homes. Deterministic proxies — always-on context size, gate wall time, gate output size — are reproducible per commit and belong in CI as budget checks (§9.4): a change that regresses them turns the gate red. Outcome efficiency — tokens-to-green-gate, wall time to a landed change, human interventions per change, pass rate — is stochastic and model-bound, so it belongs in a periodic A/B harness: the same fixed task set driven with and without the framework, several runs per task, reporting cost and outcome distributions. No off-the-shelf benchmark measures a governance framework's overhead; the closest usable substrates are long-horizon suites (SWE-bench Pro-class) and terminal-agent suites (Terminal-Bench-class) with cost reporting, adapted to run against a Leitwerk-governed copy of the target repository.

13Limitations & open questions

Leitwerk is a synthesis of what the 2023–2026 record shows works; several load-bearing pieces are nonetheless under-proven, and the framework should be adopted with that in view.

  • Bidirectional refinement is the least-charted part. Most tools avoid it by marking generated code read-only. Leitwerk's "surface drift, human reconciles" model is deliberately conservative, but reverse-deriving a spec from a hand-edit is error-prone and unproven at scale — this is where the design is most exposed.
  • Spec-completeness becomes the ballgame. If humans never read code, the only human-anchored ground truth is the spec and the behavioral review. Under-specified behavior is invisible until it fails.
  • Drift-anchor mechanism is unsettled (AST fingerprint vs. spec-id comments vs. test-name convention) and determines drift-sensor precision.
  • Agents remain weak at system-wide architecture — the framework relies on humans for decomposition and seams (P9); it does not remove that dependency, it channels it.
  • The threat model is narrower than the guarantee language suggests. Mechanical protection covers the three policy files; the check scripts are agent-editable (at T2, under review), the Claude Stop hook verifies at a configured tier rather than the diff-derived one, hooks themselves can be disabled by a user, and the soft review panel is same-model judgment that can share the generator's blind spots. The deliberate backstop is diff-tiered CI on a protected branch plus human review of policy and checks — but an adversarial analysis of the whole loop (including prompt injection through specs and reverse-derived INFERRED content) has not been done.
  • Artifact lifecycle over years is unproven. Specs, plans, and decisions-of-record accumulate monotonically; without curation the "durable artifacts" layer becomes its own context liability. The implementation now distinguishes living contracts from perishable change records (a spec's Status: line; plans archived after landing), but the cadence and mechanics of consolidation — merging landed change-specs into area specs, pruning the active set — are a design sketch, not yet practiced governance.
  • Long-lived AI-authored projects are too new for longitudinal evidence; the maintenance claims rest on 2025–2026 practice and preprints, not multi-year studies.
  • Part of the evidence base is perishable by design. The model-capability snapshots in §2 (security-failure rates, the context-rot threshold, iteration-degradation, the productivity delta) were measured on GPT-4-class or early-2025 models; they are dated in §14 and meant to be re-checked. As long-context reliability and secure-by-default behaviour improve, the necessity of P4 (retrieve, don't preload) and the intensity of the security gate should be re-tuned — the framework treats this as scheduled maintenance of its own assumptions, not a fixed liability.

These are tracked as the questions Phases 0–1 must answer with evidence rather than assumption.
